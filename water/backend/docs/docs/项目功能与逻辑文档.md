## 1. 项目概述

### 1.1 项目背景与目标

海水养殖区块链水质监管平台是一个面向海水养殖行业的综合性监管系统。平台通过区块链技术实现水质数据、养殖记录、企业资质等关键数据的可信存证与不可篡改存储，结合 IoT 设备自动采集与人工上报双通道数据录入、AI 智能助手辅助决策，构建"数据采集 - 审核 - 上链 - 监管"全链路闭环。

**核心目标**：
- 实现养殖全流程数据（水质、苗种、投喂、用药、收获）的数字化管理
- 通过区块链存证确保关键数据的真实性与不可篡改性
- 建立养殖户 - 监管局 - 管理员三级角色体系，实现分级审核与监管
- 提供实时水质异常预警机制
- 支持 AI 智能问答辅助

### 1.4 全局约定

- **响应格式**：统一使用 `BaseResponse<T>` 包装，包含 code、message、data
- **错误码体系**：`ErrorCode` 枚举定义全局错误码（0=成功，4xxxx=客户端错误，5xxxx=服务端错误，6xxxx=文件错误，7xxxx=区块链错误）
- **认证方式**：请求头 `satoken` 传递 token，Sa-Token 管理会话
- **权限校验**：`@AuthCheck(roleType = UserRole.XXX)` 注解 + `AuthInterceptor` AOP 切面
- **上链并发控制**：`@Lock(lockClass = Client.class)` 注解 + `LockAspect` AOP 切面，使用 `synchronized(Client.class)` 保证上链操作串行
- **逻辑删除**：MyBatis-Plus 全局配置 `isDelete` 字段，删除值=1，未删除值=0
- **密码加密**：MD5 + salt 方式（`PasswordUtil.encrypt` / `PasswordUtil.verify`），支持明文密码自动迁移
- **ID 生成**：雪花算法（`IdUtil.getSnowflakeNextId()`）用于 userId，数据库自增用于其他表

## 2. 功能模块总结

### 2.1 管理员（Admin）功能

| 功能 | 说明 |
|------|------|
| 系统初始化 | 应用启动时自动创建 admin/123456 管理员账户，并生成区块链密钥对 |
| 用户管理 | 查看用户列表（分页）、创建用户、修改用户信息/状态、删除用户、批量操作 |
| 监管局审批 | 查看监管局注册审核申请列表、批准/拒绝审核 |
| 公告管理 | 发布公告（指定目标角色）、查看公告列表、删除公告（已上链不可删）、上链公告 |
| 数据看板 | 查看用户统计、上链数量、许可证统计、公告数量等 |
| Excel 导出 | 导出用户列表 |

### 2.2 养殖户（Farmers）功能

| 功能 | 说明 |
|------|------|
| 注册/登录 | 注册需验证码，自动启用（userStatus=1），自动生成区块链密钥对 |
| 绑定管理 | 选择监管局并申请绑定、查看绑定状态、主动解绑 |
| 养殖池管理 | 创建/更新/删除养殖池，查看自己的养殖池列表 |
| 苗种投放 | 创建/修改/删除苗种投放记录（仅待审核状态可修改/删除） |
| 投喂记录 | 创建/修改/删除投喂记录 |
| 用药记录 | 创建/修改/删除用药记录 |
| 收获记录 | 创建/修改/删除收获记录 |
| 排污许可证 | 上传许可证图片并提交、查看自己的许可证、删除被拒绝的许可证 |
| 水质数据 | 手动上报水质数据（需审核）、查看水质数据分页列表、删除未上链数据 |
| 通知 | 查看通知列表、未读数、标记已读/全部已读、删除 |
| 数据看板 | 查看自己的养殖池/水质/审核统计 |
| AI 助手 | SSE 流式对话 |

### 2.3 监管局（Manager）功能

| 功能 | 说明 |
|------|------|
| 注册/登录 | 注册需验证码，注册后禁用（userStatus=0），需提交审核申请经管理员审批通过后启用 |
| 审核申请 | 提交审核申请（机构名称、管辖区域等）、查看审核状态 |
| 绑定管理 | 查看绑定申请列表、批准/拒绝绑定、主动解除绑定、上链绑定记录 |
| 养殖池审核 | 查看管辖养殖户的养殖池、审核/批量审核养殖池、上链养殖池 |
| 养殖记录审核 | 审核苗种/投喂/用药/收获记录，批量审核，上链 |
| 排污许可证审核 | 审核许可证（通过自动上链）、查看所有许可证、上链验证 |
| 水质数据审核 | 审核手动上报的水质数据、批量审核 |
| 预警阈值配置 | 创建/修改/查看预警阈值（水温、盐度、pH、溶解氧、氨氮、亚硝酸盐） |
| Excel 导出 | 导出养殖池、水质数据、苗种、投喂、用药、收获记录 |
| 数据看板 | 查看管辖养殖户统计、水质异常率、7日趋势、各类型审核统计 |
| 通知 | 同养殖户 |
| AI 助手 | 同养殖户 |

### 2.4 通知系统

| 通知类型 | 触发场景 |
|----------|---------|
| AUDIT（审核通知） | 养殖池/苗种/投喂/用药/收获/水质数据/许可证审核结果 |
| WARNING（预警通知） | 水质数据超出阈值（同时通知养殖户和监管局） |
| ANNOUNCEMENT（公告通知） | 管理员发布公告时按角色群发 |
| BINDING（绑定通知） | 绑定申请/通过/拒绝/解绑事件、监管局注册审批结果 |

## 6. 服务层逻辑详解

### 6.1 UserService（用户服务）

**核心职责**：用户注册、登录、管理员 CRUD、密码管理

| 方法 | 说明 | 事务 | 关键逻辑 |
|------|------|------|---------|
| `init()` | 系统启动自动创建管理员 | 否 | `@PostConstruct`，检查 admin 账户是否存在，不存在则创建并生成区块链密钥对 |
| `registerFarmers(req)` | 养殖户注册 | `@Transactional` | 验证码校验 → `synchronized(account.intern())` 防并发 → 账号唯一性校验 → 创建用户（userStatus=1）→ 生成区块链密钥对 → 自动登录 |
| `registerManager(req)` | 监管局注册 | `@Transactional` | 同上，但 userStatus=0（需审核） |
| `loginAdmin/Farmers/Manager(req)` | 分角色登录 | 否 | 按角色查用户 → 密码校验 → 明文密码自动迁移加密 → 状态检查 → Sa-Token 登录 |
| `updateProfile(req)` | 修改个人信息 | 否 | 支持修改姓名、密码（需旧密码验证）、extInfo（按角色校验 JSON 结构） |
| `adminListUsers(pageNum, pageSize)` | 分页查用户 | 否 | MyBatis-Plus 分页查询 |
| `adminUpdateStatus(req)` | 修改用户状态 | 否 | userStatus 必须为 0 或 1 |
| `adminUpdateUser(req)` | 修改用户信息 | 否 | 修改账号时检查唯一性 |
| `adminCreateUser(req)` | 管理员创建用户 | `@Transactional` | `synchronized` 防并发，校验角色类型，生成密钥对，userStatus=1 |
| `adminDeleteUser(userId)` | 删除用户 | 否 | 禁止删除管理员账户 |
| `batchUpdateStatus(req)` | 批量修改状态 | 否 | 跳过管理员 |
| `batchDeleteUsers(req)` | 批量删除 | 否 | 跳过管理员 |

**并发控制**：注册和管理员创建用户均使用 `synchronized(req.getUserAccount().intern())`，基于字符串常量池实现账号级别的互斥锁，防止同一账号重复注册。

### 6.2 ManagerAuditService（监管局审核服务）

**核心职责**：监管局注册后的审核申请流程

| 方法 | 说明 | 事务 | 关键逻辑 |
|------|------|------|---------|
| `submit(req)` | 提交审核申请 | 否 | 校验无待审核申请 → 校验无已通过申请 → 创建记录 status=0 |
| `getStatus()` | 查看审核状态 | 否 | 查最新一条记录 |
| `adminList(status, pageNum, pageSize)` | 管理员查看列表 | 否 | 支持按状态筛选，分页 |
| `approve(req)` | 批准审核 | `@Transactional` | 更新审核记录 status=1 → **启用用户** userStatus=1 → 发送 BINDING 通知 |
| `reject(req)` | 拒绝审核 | 否 | 更新审核记录 status=2 → 发送 BINDING 通知 |
| `upChain(id)` | 上链审批记录 | `@Lock` | 校验 status=1 → 通过 ChainStoreService 上链 |

**关键逻辑**：`approve` 方法同时更新 `manager_audit` 和 `user` 两张表，需事务保证一致性。

### 6.3 ManagerFarmerService（绑定管理服务）

**核心职责**：养殖户与监管局的绑定关系管理

| 方法 | 说明 | 事务 | 关键逻辑 |
|------|------|------|---------|
| `listManagers()` | 获取监管局列表 | 否 | 查 role=manager 且 userStatus=1 的用户 |
| `apply(req)` | 养殖户申请绑定 | 否 | 校验目标用户是监管局 → 校验无未拒绝的绑定（status!=2）→ 创建记录 status=0 → 通知监管局 |
| `getStatus()` | 查看绑定状态 | 否 | 查最新一条记录，填充监管局名称 |
| `managerList()` | 监管局查看绑定 | 否 | 按 managerId 查询 |
| `approve(id)` | 同意绑定 | 否 | 校验归属+状态 → status=1 → 通知养殖户 |
| `reject(id)` | 拒绝绑定 | 否 | 校验归属+状态 → status=2 → 通知养殖户 |
| `validateFarmerBound(farmerId)` | 校验绑定（公共方法） | 否 | 查 status=1 的绑定记录，不存在则抛异常 |
| `getManagedFarmerIds(managerId)` | 获取管辖养殖户ID列表（公共方法） | 否 | 查所有 status=1 的绑定记录 |
| `validateManagerFarmer(managerId, farmerId)` | 校验管辖关系（公共方法） | 否 | 校验 managerId 与 farmerId 存在 status=1 的绑定 |
| `unbind()` | 养殖户解绑 | 否 | 查 status=1 记录 → 逻辑删除 → 通知监管局 |
| `managerUnbind(id)` | 监管局解绑 | 否 | 校验归属+status=1 → 逻辑删除 → 通知养殖户 |
| `upChain(id)` | 上链绑定记录 | `@Lock` | 校验 status=1 → 序列化绑定信息 JSON → ChainStoreService 上链 |

**公共方法调用链**：`validateFarmerBound` 被几乎所有养殖户操作调用（创建养殖池、创建各类记录等），`getManagedFarmerIds` 和 `validateManagerFarmer` 被所有监管局查询/审核操作调用。

### 6.4 PondService（养殖池服务）

**核心职责**：养殖池 CRUD、审核、上链

| 方法 | 说明 | 事务 | 关键逻辑 |
|------|------|------|---------|
| `create(req)` | 创建养殖池 | 否 | 校验绑定 → 创建记录 auditStatus=0 |
| `list()` | 查询自己的养殖池 | 否 | 校验绑定 → 按 userId 查询 |
| `update(req)` | 更新养殖池 | 否 | 校验绑定+归属 → 局部更新 → **若已审核通过则重置为待审核**（auditStatus=0） |
| `updateStatus(req)` | 更新池塘状态 | 否 | 校验绑定+归属 → 更新 status 字段 |
| `delete(id)` | 删除养殖池 | 否 | 校验绑定+归属 → 逻辑删除 |
| `managerList(...)` | 监管局查看 | 否 | 获取管辖 farmerIds → 分页查询，支持 farmerId/auditStatus 筛选 |
| `audit(req)` | 单条审核 | 否 | 校验管辖 → 更新 auditStatus → AUDIT 通知 |
| `batchAudit(req)` | 批量审核 | `@Transactional` | 遍历 id 列表逐条审核 |
| `upChain(ids)` | 上链 | `@Lock` + `@Transactional` | 校验管辖+审核通过 → ChainStoreService 上链 |

**特殊逻辑**：养殖池的 update 不受 auditStatus 限制（与养殖记录不同），但如果已审核通过则修改后重置为待审核。

### 6.5 SeedRecordService（苗种投放服务）

**核心职责**：苗种投放记录的 CRUD、审核、上链

| 方法 | 说明 | 事务 | 关键逻辑 |
|------|------|------|---------|
| `create(req)` | 创建记录 | 否 | 校验绑定 + 校验养殖池归属 → auditStatus=0 |
| `list(pondId, pageNum, pageSize)` | 分页列表 | 否 | 校验绑定 → 按 userId 查询，支持 pondId 筛选 |
| `managerList(...)` | 监管局查看 | 否 | 获取管辖 farmerIds → 分页查询 |
| `audit(req)` | 单条审核 | 否 | 校验管辖 → 更新 auditStatus → AUDIT 通知 |
| `batchAudit(req)` | 批量审核 | `@Transactional` | 遍历审核 |
| `update(req)` | 修改记录 | 否 | 校验绑定+归属 → **auditStatus 必须为 0** → 局部更新 |
| `delete(id)` | 删除记录 | 否 | 校验绑定+归属 → **auditStatus 必须为 0** → 删除 |
| `upChain(ids)` | 上链 | `@Lock` + `@Transactional` | 校验管辖+auditStatus=1 → ChainStoreService 上链 |

**辅助方法** `validatePondOwner(pondId, userId)`：校验养殖池存在且属于当前用户。

### 6.6 FeedRecordService（投喂记录服务）

结构与 SeedRecordService 完全一致，区别在于字段不同（feedDate, feedBrand, feedAmount 等）。所有方法模式相同：创建→审核→修改→删除→上链。

### 6.7 MedicineRecordService（用药记录服务）

结构与 SeedRecordService 完全一致，区别在于字段不同（medicineDate, medicineName, purpose, dosage, withdrawalStart, withdrawalEnd 等）。

### 6.8 HarvestRecordService（收获记录服务）

结构与 SeedRecordService 完全一致，区别在于字段不同（harvestDate, batchNo, spec, totalWeight, survivalRate, buyerInfo, qualityResult 等）。

### 6.9 WaterService（水质与许可证服务）

**核心职责**：水质数据管理（IoT采集/手动上报/上链）、排污许可证管理

| 方法 | 说明 | 事务 | 关键逻辑 |
|------|------|------|---------|
| `commit(req)` | 提交许可证 | 否 | 校验无 status=0 或 status=1 的许可证 → 创建 status=0 |
| `verify(req)` | 审核许可证 | `@Lock` + `@Transactional` | 校验 status=0 → 更新状态 → **通过则自动上链**（WaterRawCall.uploadCompanyCert）→ AUDIT 通知 |
| `isOnChain(id)` | 链上比对 | 否 | 从链上读取 CompanyCert 列表 → 与数据库对比 |
| `companyQuery()` | 养殖户查自己的许可证 | 否 | 按 userId 查询，填充企业名称 |
| `managerAllQuery(status)` | 监管局查所有许可证 | 否 | 支持按状态筛选，填充企业名称 |
| `deleteCert(certId)` | 删除许可证 | 否 | 校验归属 → **仅 status=2 可删** |
| `gen(req)` | 生成测试数据 | 否 | 生成随机数据 |
| `collect(req)` | IoT 设备数据采集 | 否 | 自动阈值检测 → 异常则双向通知 → auditStatus=1 |
| `manual(req)` | 手动上报水质数据 | 否 | 校验绑定 + 校验养殖池归属 → 阈值检测 → 异常则双向通知 → auditStatus=0 |
| `page(dataType, pageNum, pageSize)` | 分页查询 | 否 | 按 userId + dataType 查询 |
| `upChain(req)` | 水质数据上链 | `@Lock` + `@Transactional` | 校验未上链 → **AES 加密 data 字段** → WaterRawCall.insertWaterData → 标记 isOnChain=true |
| `managerDataList(...)` | 监管局查看水质数据 | 否 | 获取管辖 farmerIds → 分页查询 |
| `auditWaterData(req)` | 审核水质数据 | 否 | 校验管辖 + auditStatus=0 → 更新 → 通知 |
| `batchAuditWaterData(req)` | 批量审核 | `@Transactional` | 遍历审核 |
| `deleteWaterData(dataId)` | 删除水质数据 | 否 | 校验归属 + 未上链 |
| `step1Info()` | 水质步骤1信息 | 否 | 查询同账号下的养殖户和监管局信息 |

**两个数据采集通道的区别**：
- `collect`（IoT）：无需认证，auditStatus 默认为 1（已通过），自动预警
- `manual`（手动）：需认证+绑定+养殖池归属，auditStatus=0（待审核），自动预警

### 6.10 WarningThresholdService（预警阈值服务）

**核心职责**：预警阈值的配置与异常判断

| 方法 | 说明 | 事务 | 关键逻辑 |
|------|------|------|---------|
| `createThreshold(req)` | 新建阈值配置 | 否 | 每个 managerId 只能创建一次 |
| `updateThreshold(req)` | 修改阈值配置 | 否 | 仅更新传入的非 null 字段 |
| `getThreshold()` | 获取阈值配置 | 否 | 按 managerId 查询 |
| `isAbnormal(dataType, value, managerId)` | 判断是否异常（公共方法） | 否 | 按 dataType 匹配 min/max → value < min 或 value > max 则为异常 |

**dataType 与阈值字段映射**：
- 2（水温）→ waterTempMin/Max
- 3（盐度）→ salinityMin/Max
- 4（pH）→ phMin/Max
- 5（溶解氧）→ doMin/Max
- 6（氨氮）→ nh3nMin/Max
- 7（亚硝酸盐）→ no2Min/Max
- 0/1（IoT 数据）→ 不做阈值检测

### 6.11 AnnouncementService（公告服务）

## 7. 前端对接实战注意事项

### 7.1 认证与鉴权

**Token 传递方式**：
- 所有需要认证的接口，必须在请求头（Header）中携带 `satoken` 字段
- 示例：`satoken: xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`
- Token 有效期 86400 秒（24小时），活跃超时同样为 86400 秒
- 每次登录生成新 token（`is-share: false`），允许同一账号多地登录（`is-concurrent: true`）

**权限校验机制**：
- 后端通过 `@AuthCheck(roleType = "admin"|"farmers"|"manager")` 注解校验
- 前端需根据用户角色控制页面/按钮的显示与隐藏
- 注意：`/water/data/collect`（IoT 采集）和 `/user/captcha`、`/common/indexData` 不需要 token

**登录流程**：
1. 调用 `/user/captcha` 获取验证码（返回 captchaKey 和 base64 图片）
2. 用户输入账号、密码、验证码
3. 根据角色调用对应登录接口（`/user/login/admin`、`/user/login/farmers`、`/user/login/manager`）
4. 返回 token 和用户信息，前端存储 token
5. 调用 `/user/getLoginInfoByToken/{token}` 刷新用户信息

### 7.2 统一响应格式

```json
{
  "code": 0,
  "message": "ok",
  "data": { ... }
}
```

- `code=0` 表示成功
- `code!=0` 时 `message` 包含错误描述
- 分页响应结构：`{ "total": 100, "current": 1, "size": 10, "data": [...] }`

### 7.3 角色状态管理注意事项

**监管局注册后的特殊流程**：
1. 注册成功返回 token，但 userStatus=0
2. 前端应引导用户填写审核申请（`/manager-audit/submit`）
3. 提交后可轮询 `/manager-audit/status` 查看审核状态
4. 管理员审批通过后，监管局才能正常使用系统
5. 前端应在检测到 userStatus=0 时限制大部分功能，仅允许查看审核状态

**养殖户注册后的绑定要求**：
1. 注册成功后可登录，但需绑定监管局才能进行养殖操作
2. 前端应在检测到无绑定关系时引导用户进行绑定
3. 所有养殖操作（养殖池、记录等）在未绑定时会返回 `40101 无权限` 错误

### 7.4 审核状态交互

**养殖记录（苗种/投喂/用药/收获）**：
- 创建后 auditStatus=0（待审核），此时可修改、删除
- 审核后 auditStatus=1（通过）或 2（拒绝），不可修改/删除
- 前端应根据 auditStatus 控制修改/删除按钮的显示

**养殖池**：
- 创建后 auditStatus=0
- 已通过（auditStatus=1）的养殖池仍可修改，但修改后自动重置为待审核
- 前端应提示用户"修改后需重新审核"

**水质数据**：
- IoT 采集的数据 auditStatus=1（已通过），无需审核
- 手动上报的数据 auditStatus=0，需监管局审核
- 已上链的数据（isOnChain=true）不可删除

### 7.5 文件上传

- 接口：`POST /common/upload`，`multipart/form-data` 格式
- 最大文件大小：100MB
- 返回格式：`{ "url": "完整URL", "fileName": "相对路径", "newFileName": "文件名", "originalFilename": "原始文件名" }`
- 许可证上传流程：先调用上传接口获取 imageUrl → 再调用提交接口

### 7.6 AI 助手对接

- 接口：`GET /ai/chat?message=xxx`
- 需要在 Header 中传递 `satoken`
- 返回 SSE（Server-Sent Events）流式数据
- 前端使用 `EventSource` 或类似库接收
- 每条消息格式：`data: "内容片段"`

### 7.7 数据类型映射

前端展示水质数据时需映射 dataType：

| dataType | 显示名称 |
|----------|---------|
| 0 | TDS |
| 1 | 浑浊度 |
| 2 | 水温 |
| 3 | 盐度 |
| 4 | pH |
| 5 | 溶解氧 |

### 7.8 通知轮询建议

- 建议前端定时轮询 `GET /notification/unreadCount`（如每 30 秒）
- 在页面头部显示未读数 badge
- 收到 WARNING 类型通知时应做特殊高亮或弹窗提醒

### 7.9 导出功能

- 所有导出接口直接返回文件流（`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`）
- 前端使用 `window.open(url)` 或创建 `<a>` 标签下载
- 需要在请求中携带 satoken header（建议使用 axios + responseType: 'blob'）

### 7.10 分页参数约定

- `pageNum`：页码，从 1 开始
- `pageSize`：每页条数
- 部分接口使用 `@RequestParam` 接收，部分使用请求体

### 7.11 常见错误码处理

| 错误码 | 含义 | 前端处理建议 |
|--------|------|-------------|
| 40000 | 请求参数错误 | 检查表单数据 |
| 40001 | 请求头参数错误 | 检查 satoken header |
| 40100 | 未登录 | 跳转登录页 |
| 40101 | 无权限 | 提示用户权限不足 |
| 40400 | 数据不存在 | 提示并刷新列表 |
| 50001 | 操作失败 | 显示具体错误信息 |
| 70003 | 区块链调用失败 | 提示上链失败，建议重试 |

### 7.12 区块链看板

- 接口：`GET /common/indexData`
- 依赖 WeBASE-Front 服务正常运行
- 返回数据：节点数、区块数、交易总数、节点状态列表

### 7.13 用户合约绑定

- 前端可通过 `POST /user_contract/create` 绑定合约地址
- 合约类型：`"水质"` 对应 Water.sol，`"键值存储"` 对应 KeyValueStore.sol
